# 프로모션 임시 패스 {#promotional-temp-pass}

>[!NOTE]
>
>이 페이지의 컨텐츠는 정보용으로만 제공됩니다. 이 API를 사용하려면 Adobe의 현재 라이선스가 필요합니다. 무단 사용이 허용되지 않습니다.

## 기능 요약 {#feature-summary}

프로그래머에게는 MVPD가 있는 계정 자격 증명이 없는 사용자에게 보호된 콘텐츠에 임시 액세스 권한을 제공할 수 있습니다.

프로모션 임시 패스는 사용자가 프로그래머에게 유효한 ID 정보(예: 이메일 주소)를 제공한 후 프로모션 캠페인을 실행하는 데 사용하도록 설계되었습니다 **사전 정의된 시간 동안 다른 VOD 제목 수**.

>[!IMPORTANT]
>
>Adobe은 PII(개인 식별 정보)를 저장하지 않습니다. 따라서 프로그래머는 Primetime 인증 API에 대한 정보를 제공한 고유 사용자에 대해 해시를 설정해야 합니다.

프로모션 임시 패스는 [임시 통과](/help/authentication/temp-pass.md) 모든 Temp Pass 기능이 포함되어 있음을 의미합니다.

사전 정의된 최대 VOD 제목 수나 사전 정의된 시간이 초과되면 Adobe Primetime 인증 서버에서 인증 토큰이 지워질 때까지 동일한 장치의 컨텐츠에 액세스하거나 동일한 사용자 식별자 정보(예: 이메일 주소)를 사용하여 액세스할 수 없습니다.

>[!NOTE]
>
>임시 패스는 Premium 워크플로우 패키지의 일부입니다. 이 기능을 사용하려면 Primetime 영업 담당자에게 문의하십시오.

## 임시 통과 및 프로모션 임시 통과 비교 {#tp-ptp-comparison}

| 임시 통과 | 프로모션 임시 패스 |
|----------------------------------|----------------------------------------------------------------------------------------|

## 기능 세부 사항 {#feature-details}

이 기능을 사용하면 사용자가 프로그래머 애플리케이션에서 전자 메일 주소와 같은 고유한 정보를 제공한 후 특정 장치(전화 및 태블릿)에서 프로모션 컨텐츠에 액세스할 수 있습니다.

프로그래머는 인증 및 인증 API에 대한 사용자의 PII에 대한 해시를 제공합니다. 이 해시는 사용자 및 장치를 식별하는 고유 키를 생성할 때 장치 ID와 함께 사용됩니다.

장치 ID 및 사용자가 제공한 정보와 아래 논리를 따라 Adobe Primetime 인증이 사용자가 새 평가판에 있는지 아니면 기존 평가판에 있는지 결정합니다.

* 사용자가 제공한 정보에 대한 새 해시(예: 이메일), 새 장치 ID => 새 평가판
* 사용자가 제공한 정보에 대한 기존 해시(예: 이메일), 새 장치 ID => 기존 평가판(기존 해시가 사용자 제공 정보에 대해 있음(예: 이메일))
* 사용자가 제공한 정보에 대한 새 해시(예: 이메일), 기존 장치 Id => 기존 평가판(기존 장치 Id 사용)
* 사용자가 제공한 정보에 대한 기존 해시(예: 이메일), 기존 장치 Id => 기존 평가판

>[!NOTE]
>사용자가 제공한 정보의 유효성 검사 및 해시는 Adobe이 아니라 프로그래머가 처리합니다.

**프로모션 임시 패스 기능은 다음 속성을 기반으로 구성할 수 있습니다.**

* 사용자가 제공한 정보 키(예: 이메일)
* 사용자가 사용할 수 있는 리소스 수
* TTL - 사용자가 구성된 리소스 수를 사용할 수 있는 기간입니다

### 사용자 메타데이터 {#user-metadata}

Programmer의 응용 프로그램을 쉽게 구현하려면 다음과 같이 하십시오 **사용자 메타데이터 정보가 노출됨** 프로모션 임시 패스에서 해당 키(키를 활성화하려면 tve-support@adobe.com에 문의)가 있습니다.

* **maintain_resources**: 현재 사용자가 사용할 수 있는 나머지 리소스 수
* **used_assets**: 현재 사용자가 이미 사용한 리소스 목록
* **expiration_date**: 현재 사용자의 만료 날짜

### 보기 시간은 어떻게 계산됩니까? {#compute-viewing-time}

Temp Pass가 유효한 상태로 남아 있는 시간은 사용자가 Programmer의 응용 프로그램에서 콘텐츠를 보는 데 걸리는 시간과 관련이 없습니다. 홍보 임시 패스를 통해 인증을 위한 초기 사용자 요청이 있을 때, 초기 현재 요청 시간을 프로그래머가 지정한 TTL(기간 기간 기간 기간 기간)에 추가하여 만료 시간을 계산합니다.

### 인증 및 인증 {#authn-authz}

프로모션 임시 통과 플로우의 경우 인증 및 인증이 실제 MVPD와 통신하지 않습니다. **따라서 모든 권한 부여 요청이 성공합니다** 이러한 모든 조건이 충족되는 한:

* 지정된 리소스에 대해 인증 토큰이 유효합니다.
* 번호 **used_assets** 은 프로그래머가 설정한 제한보다 낮습니다.
* **expiration_date** 값은 현재 날짜 이후입니다.

### 프리플라이트 동작 {#preflight-beh}

Inpromotion Temp Pass MVPD에 대해 프리플라이트 또는 사전 권한 부여 요청이 수행되면 반환되는 해당 프리플라이트 응답에는 프리플라이트 요청의 전체 리소스 목록이 프리플라이트 성공한 것으로 포함됩니다.

이것의 이면에 있는 논리는 다음과 같습니다. 판촉 임시 통과 권한 조건은 특정 자원이 아닌 시간 및 자원 번호 제한을 기반으로 합니다. 특히 시간 제한이 충족되고 리소스 제한을 초과하지 않는 한 호출된 리소스가 승인됩니다.

### SSO {#sso}

SSO가 &quot;요청자당 인증&quot;이 활성화된 상태에서 구성된 프로모션 임시 패스의 인스턴스에 대해 활성화되지 않습니다. 즉, 사용자가 애플리케이션 A에서 동일한 프로모션 임시 패스와 통합된 다른 애플리케이션 B로 전환하면 사용자는 자동으로 로그인되지 않습니다.

### 로그아웃 {#logout}

장치의 모든 토큰은 로그아웃 시 삭제됩니다. 따라서 프로모션 임시 패스에서 일반 사용자가 선택한 MVPD로 전환하는 것은 이 구현에 의존하지 않아야 합니다. 권장 사항은 `setSelectedProvider(null)` 를 사용하여 애플리케이션 상태를 지우고 인증 흐름을 다시 시작하여 사용자 경험이 향상됩니다.

## 프로모션 임시 패스 구현 {#impl-promo-tempass}

프로모션 Temp Pass에는 다음과 같은 클라이언트측 기능이 필요합니다.

* **사용자 식별자 정보(예: 전자 메일 주소 전파)** (인증 및 인증 플로우에서 사용자의 이메일 주소 전송). 전자 메일은 인증 및 인증 토큰을 바인딩하기 위해 Adobe Primetime 인증에 필요합니다(와 유사함) `device_ID`, 모든 호출에 필요).
* **강제 인증** - 사용자가 이미 인증되었을 때 Programmer가 인증 흐름을 강제로 수행하도록 허용 사용자 메타데이터(사용자 메타데이터 키)를 강제로 새로 고치려면 이 기능이 필요합니다 **used_assets** 는 앱이 시작될 때마다 사용 가능한 리소스 수를 포함합니다. 사용자가 여러 장치에서 로그인할 수 있으므로 앱 시작 중에 장치에 있는 사용자 메타데이터는 신뢰할 수 없으므로 해당 특정 사용자(전자 메일 주소로 식별됨)의 현재 상태를 반영하려면 업데이트해야 합니다.


>[!IMPORTANT]
>강제 인증은 iOS 및 Android에서만 가능합니다.
>X분이 지난 후 Primetime 인증에는 무료 스트리밍을 중지하는 기본 메커니즘이 없습니다. Primetime 인증 발행 중단 **권한** 및 **짧은 미디어** 사용자가 Y의 자유 리소스를 소비하면 토큰이 표시됩니다. 프로모션 임시 패스가 만료되면 프로그래머가 액세스 권한을 제한합니다.

## 보안 {#security}

>[!IMPORTANT]
>Adobe은 PII(개인 식별 정보)를 저장하지 않습니다. 따라서 프로그래머는 Primetime 인증 API에 대한 정보를 제공한 고유 사용자에 대해 해시를 설정해야 합니다.

**사용자 식별자 정보 해시**

Adobe은 **SHA-2** 가족 또는 특정 **SHA-256**, **SHA-512** 함수는 Adobe으로 전송되기 전에 데이터에 대해 작동합니다.

예, **SHA-256** over **&quot;user@domain.com&quot;** is **&quot;f7ee5ec7312165148b69fcca1d29075b14b8aef0b5048a332b18b88d09069fb7&quot;**.

## 프로모션 임시 패스 재설정 또는 제거 {#reset-promo-tempass}

특정 비즈니스 규칙에서는 프로모션 임시 패스를 정기적으로 제거해야 합니다. 이를 위해 Primetime 인증은 프로그래머에게 *공용* 웹 API, 아래에 설명되어 있습니다.

| `DELETE https://mgmt.auth.adobe.com/reset-tempass/v2/reset` |
|----|
| <ul><li>프로토콜: **https**</li><li>호스트:<ul><li>릴리스: **mgmt.auth.adobe.com**</li><li>사전 요구 사항: **mgmt-prequal.auth.adobe.com**</li></ul></li><li>경로: **/reset-tempass/v2/reset**</li><li>쿼리 매개 변수: **device_id=all&amp;requestor_id=THE_REQUESTOR_ID&amp;mvpd_id=THE_TEMPASS_MVPD_ID**</li><li>헤더: Api 키: **1232293681726481**</li> <li>응답:<ul><li>성공: **HTTP 204**</li><li>실패: **HTTP 400** 잘못된 요청의 경우 **HTTP 401** apiKey가 지정되지 않은 경우 **HTTP 403** apiKey가 잘못된 경우</li></ul></li></ul> |

Temp Pass를 삭제하기 위한 요구 사항 외에 In판촉 Temp Pass는 다음과 같이 전송된 사용자 ID 정보를 통해 해시를 사용합니다 **generic_data** 인증 및 삭제 권한 부여

전체 JSON이 아닌 해시가 전송됩니다.

```cURL
$ curl -X DELETE -H "Authorization:Bearer H4j7cF3GtJX81BrsgDa10GwSizVz" "https://mgmt.auth.adobe.com/reset-tempass/v2.1/reset/generic?key=f7ee5ec7312165148b69fcca1d29075b14b8aef0b5048a332b18b88d09069fb7&requestor_id=REF&mvpd_id=FlexibleTempPass"
```

## 제한 사항 {#limitations}

이 섹션에서는 프로모션 임시 패스의 현재 구현에 적용되는 제한 사항에 대해 설명합니다.

### Clientless {#lim-clientless}

**고유 장치 ID가 없는 스마트 장치**

일부 스마트 장치 앱에서 고유 장치 ID를 제공할 수 있는 것은 아닙니다. Adobe Primetime 인증이 없으면 Adobe 등록 코드 서비스에서 생성한 UUID를 고유 장치 ID로 사용할 수 있습니다. 즉, 사용자가 로그아웃하면 인증 및 인증 토큰이 삭제됩니다. 사용자가 다시 인증을 시도하면 이번에는 다른 사용자 정보(예: 이메일)로 사용자가 다시 인증할 수 있습니다. Adobe은 사용자가 시스템을 &quot;바보&quot;하고 논리를 추가하여 평가판인지 기존 평가판인지 여부를 결정할 수 없는 UI 흐름을 추가할 것을 권장합니다.

**임시 패스 재설정/제거**

Xbox360 및 Xbox One의 특정 경우에는 Temp Pass를 재설정할 수 없습니다. 이러한 플랫폼에서는 Reset Temp Pass 도구에서는 추가적인 장치 ID 구문 분석이 필요하므로 사용할 수 없습니다.
