# Adobe Analytics에 Primetime 인증 서버 측 데이터 통합

>[!NOTE]
>
>이 페이지의 컨텐츠는 정보용으로만 제공됩니다. 이 API를 사용하려면 Adobe의 현재 라이선스가 필요합니다. 무단 사용이 허용되지 않습니다.

Adobe Primetime 인증 고객은 쉽게 소비할 수 있도록 Adobe Analytics 대시보드에서 Primetime 인증(Adobe Pass) 서버 측 데이터를 확인하려고 합니다.

이 데이터는 MVPD당 인증 전환율, MVPD 사용자 ID를 기반으로 하는 고유 사용자 등과 같은 중요한 TVE 지표를 추적하는 데 사용됩니다.

방문자 ID가 없는 경우 사용자 활동을 아래의 특정 이벤트 이상으로 추적할 수 없으므로 이미 있는 경우 클라이언트 측 구현을 대체하지 않습니다. 고객이 전달 호출에 방문자 ID를 제공하는 경우 모든 Pass 이벤트를 기존 고객 데이터와 결합할 수 있는 다른 유형의 Analytics 통합(실시간)을 잠금 해제할 수 있습니다. 여기서 이러한 새로운 유형의 가능한 통합에 대해 자세히 알아보십시오. &quot;[Primetime 인증에서 Experience Cloud ID 사용](/help/authentication/exp-cloud-id-authn.md)&quot;

## 포함된 지표 {#metrics-included-int-authn-analyt}

| 이벤트 | 설명 |
|----------------------------|----------------------------------------------------------------------------------------------------------------------|
| AuthN 요청 | 시작된 인증 흐름 수 |
| AuthN 보류 중 | 성공적으로 생성된 인증 토큰 수(클라이언트가 실제로 얻었는지 여부에 상관없이) |
| AuthN 확인 | 사용자가 성공적으로 획득한 인증 토큰 수 |
| AuthZ 요청 | 시도 권한 수 |
| AuthZ 확인 | 성공한 권한 수 |
| AuthZ 실패 | 응용 프로그램 수준에서 MVPD가 거부한 권한 수 |
| 재생 요청 | 생성된 짧은 미디어 토큰 수(재생 요청 수에 부합함) |
| 요청된 로그아웃 | 시작된 로그아웃 흐름 수 |
| 로그아웃 완료 | 완료된 로그아웃 흐름 수(성공) |
| 로그아웃 실패 | 실패한 로그아웃 흐름 수 |
| 사전 권한 부여 요청 | 시작된 사전 인증 흐름 수 |
| 사전 인증 확인 | 사전 승인된 리소스를 가진 성공적인 사전 인증 이벤트 수 |
| 사전 승인이 거부되었습니다. | 사전 승인이 거부된 리소스가 있는 사전 인증 이벤트 수 |
| 사전 권한 부여 실패 | 실패한 사전 인증 이벤트 수 |

| Adobe Analytics 이름 | 설명 |
|------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 채널 | 자격 요청을 수행하는 데 사용되는 요청자 ID입니다 |
| MVPD | 사용자에게 자격 부여 업무를 담당하는 MVPD |
| 프록시 | 프록시 MVPD(직접 통합을 위해 &quot;직접&quot;) |
| SDK 유형 | 사용된 클라이언트 SDK(Flash, HTML5, Android 기본, iOS, Clientless 등)는 |
| SDK 버전 | Adobe Primetime 인증 클라이언트 SDK의 버전 |
| 리소스 ID | 권한 부여 요청에 관련된 실제 리소스 제목(제공된 경우 MRSS 페이로드에서 항목/제목으로 추출) |
| AuthZ 오류 유형 | Adobe Primetime 인증에서 보고한 실패 이유 <br/> 다음은 가장 일반적인 값입니다 <br/> **noAuthZ** = MVPD가 사용자에게 패키지에 채널이 없다고 응답했습니다<br/> **네트워크** = MVPD에 연결할 수 없습니다(MVPD가 호출 시 문제가 발생하여 응답하지 않았습니다.)<br/> **norefreshtoken** = 이는 OAuth 구현에 엄격히 사용되며, 사용자가 암호를 변경하거나 일부 이유로 MVPD가 암호를 거부하면 발생할 수 있습니다. 일반적으로 새 인증이 발생합니다<br/> **불일치** = 인증 토큰이 있는 장치와 다른 장치에서 요청이 수행된 경우. 사용자가 시스템을 조작하려고 하지만 이러한 작업의 대부분은 장치 ID가 계산의 일부로 IP 주소를 사용하고 있었던 이전 JavaScript SDK의 컨텍스트에서 발생한 경우 발생할 수 있습니다. 사용자가 집에서 TVE를 시청한 다음 직장에서 이 오류를 트리거하고 다시 인증해야 하는 경우<br/> **잘못된** = 잘못된 요청, 누락 또는 잘못된 매개 변수<br/>  **authzNone** = 프로그래머는 특정 채널xMVPD 조합에 대한 승인을 거부할 수 있습니다. 프로그래머가 액세스 권한이 있는 백엔드 API에 의해 트리거됩니다<br/> **사기** =우리 측에는 보호 메커니즘이 있다. 사용자가 권한 부여에 실패한 다음 짧은 간격(초)으로 여러 번 다시 요청하는 경우 Adobe에서는 직접 호출을 거부합니다. 일반적으로 프로그래머가 구현에 버그가 있어 실패하면 항상 인증을 요청하는 버그가 있을 때 발생합니다. |
| 토큰 유형 | AuthZ All 및 AuthN All으로 인해 토큰이 만들어지면 성능 저하 측정에 의해 어떤 결과가 발생하는지 알고 있어야 합니다.<br/> 절차는 다음과 같습니다.<br/> &quot;normal&quot; = 일반 case<br/> &quot;authnall&quot; = AuthN All이 활성화되어 있을 때<br/> &quot;authzall&quot; = AuthZ All이 활성화되어 있을 때<br/>  &quot;hba&quot; = HBA를 사용할 때 |
| 클라이언트 없는 장치 유형 | 현재 Clientless에 사용되는 장치 플랫폼(대체 요소).<br/> 값은 다음과 같습니다.<br/> N/A - 이 이벤트는 Clientless SDK에서 발생하지 않았습니다<br/> 알 수 없음 - 장치 유형 매개 변수의 **Clientless API** 는 선택 사항이며, 값을 포함하지 않는 호출이 있습니다.<br/> 를 통해 전송된 다른 모든 값 **Clientless API**. 예를 들어 xbox, appletv 및 roku가 있습니다. |
| MVPD 사용자 ID | 쿠키 기반 방문자 ID 대체 |


## 세부 사항 {#details-int-authn-analyt}

* 지표가 삽입되고, Adobe Analytics 데이터 삽입 API를 통해 특정 보고서 세트에 이벤트별로 이벤트가 삽입됩니다
* 삽입은 30분마다 일괄 처리되어 전송됩니다. 따라서 보고서에 타임스탬프를 지정해야 합니다
* 각 고객은 하나 이상의 보고서 세트를 갖게 됩니다. 하나의 요청자 ID(채널)는 하나의 보고서 세트에만 매핑됩니다. 여러 요청자 ID는 하나의 보고서 세트에만 매핑할 수 있습니다.
* 이전 데이터를 제공할 수 있지만, 트래픽/성능 문제로 인해 특별한 주의가 필요합니다.
* 고유 방문자 변수가 MVPD 사용자 ID로 설정되어 있습니다
* 이벤트 및 eVar 매핑은 구성할 수 없습니다.
